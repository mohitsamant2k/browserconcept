🛡️ Project 2: Same-Origin Policy

See the browser block cross-origin requests in real time. Open DevTools Console (F12)!

🟢 Server A — http://localhost:3000 (You are here)
🔴 Server B — http://localhost:4000 (Different origin)

🧠 What is Same-Origin Policy (SOP)?

The browser blocks JavaScript on one origin from reading data from a different origin.
An origin = scheme + host + port

http://localhost:3000 and http://localhost:4000 are different origins (different port).
So JavaScript on Server A cannot read responses from Server B.

✅ Experiment 1: Same-Origin Fetch (ALLOWED)

Fetch data from localhost:3000/api/same-origin-data — same origin as this page. This will always work because it's the same origin.

💡 Why does this work?

The request goes from localhost:3000 → localhost:3000. Same scheme, same host, same port = same origin. No restrictions.

❌ Experiment 2: Cross-Origin Fetch (BLOCKED)

Fetch data from localhost:4000/api/secret-data — different origin! The browser will block JavaScript from reading the response.

💡 What happened?

The request was actually sent to Server B (check your terminal!). Server B did respond.
But the browser refused to give the response to JavaScript because there's no Access-Control-Allow-Origin header.
This is SOP in action — the browser enforces it, not the server.

❌ Experiment 3: Read iframe DOM from Different Origin (BLOCKED)

Load Server B's page in an iframe, then try to read its DOM content with JavaScript. The browser will block DOM access across origins.

💡 Why can't we read the iframe?

The iframe loaded fine (you can see it!), but JavaScript on this page cannot access its DOM.
Imagine if it could — any website could iframe your bank and read your account details!
SOP prevents this by blocking iframe.contentDocument across origins.

⚠️ Experiment 4: Cookies vs Same-Origin Policy (SURPRISE!)

You'd expect cookies to follow the Same-Origin Policy (scheme+host+port), right?
WRONG! Cookies use a domain+path model where ports are completely ignored.
This means localhost:3000 and localhost:4000 share the same cookie jar!

🔥 CRITICAL: Cookies ≠ Same-Origin Policy!

SOP uses: scheme + host + port → localhost:3000 ≠ localhost:4000
Cookies use: domain + path (port is IGNORED!) → localhost:3000 = localhost:4000

This means:
✅ Server A sets serverA_session → visible on BOTH :3000 and :4000!
✅ Server B sets serverB_session → visible on BOTH :3000 and :4000!
❌ Cross-origin fetch() still blocks the response (SOP), but...
⚠️ Cookies set via direct navigation are shared across ports!

🧠 Why this matters in real life:
In production, different origins have different domains (e.g., myapp.com vs api.myapp.com),
so cookies ARE isolated. But on localhost, this port-sharing is a known quirk.

🔒 Defense: Use SameSite, Secure, HttpOnly attributes, and cookie prefixes
like __Host- to lock cookies down. We'll cover this in Project 5 (Cookie Security).

⚠️ Experiment 5: Load a Script from Different Origin (ALLOWED!)

SOP does NOT block loading scripts, images, or stylesheets from other origins! This is why CDNs work — you can load jQuery from cdn.jsdelivr.net. But this is also how XSS attacks work (loading malicious scripts).

💡 SOP Exception: Resource Embedding

SOP blocks reading data cross-origin, but allows embedding resources:
✅ <script src="..."> — scripts can be loaded (but not read with fetch)
✅ <img src="..."> — images can be displayed
✅ <link href="..."> — stylesheets can be loaded
✅ <iframe src="..."> — pages can be framed (but DOM can't be read)
❌ fetch() / XMLHttpRequest — reading the response is blocked

This is an important distinction! The browser executes cross-origin scripts but won't let you read cross-origin API responses.

✅ Experiment 6: Cross-Origin Communication with postMessage (ALLOWED)

window.postMessage() is the safe, intentional way to communicate across origins. Both sides must explicitly opt-in — the sender calls postMessage() and the receiver listens with addEventListener('message').

💡 postMessage Security

postMessage works because both sides consent:
1. Sender specifies the target origin: iframe.postMessage(data, 'http://localhost:4000')
2. Receiver checks event.origin before processing the message

⚠️ Always validate the origin! Without checking, any website could send malicious messages to your iframe.
